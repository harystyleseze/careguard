# Accessibility Audit Runbook

This runbook documents WCAG 2.1 AA compliance testing for the CareGuard dashboard.

## Automated Testing

### axe DevTools Playwright

The dashboard uses `@axe-core/playwright` to catch accessibility violations in CI.

```bash
cd dashboard && npm run test:a11y
```

This runs all dashboard tabs through Playwright and reports violations.

## Manual Testing

### Keyboard Navigation

Test tab order and keyboard access without mouse:

1. Open dashboard in Firefox (better focus ring support)
2. Use `Tab` to navigate forward, `Shift+Tab` backward
3. Verify tab order follows visual left-to-right, top-to-bottom
4. Check all interactive elements are reachable (buttons, links, form fields)
5. Confirm `Enter` and `Space` work on buttons

**Critical paths:**
- Overview tab: run agent task, toggle pause
- Medications tab: view/sort medications
- Bills tab: view/audit bill items
- Policy tab: edit and save policy thresholds
- Wallet tab: view balance
- Activity tab: pagination, clear logs
- Settings tab: toggle agent pause, configure policy

### Screen Reader Testing

Test with VoiceOver on Safari (macOS):

```bash
# Enable VoiceOver
Cmd + F5  # or System Preferences > Accessibility > VoiceOver

# Navigation
VO + Right Arrow     Navigate forward
VO + Left Arrow      Navigate backward
VO + U               Open rotor (headings, form controls, etc.)
VO + Space           Activate button/link
```

**Checklist:**
- [ ] Page title is announced ("CareGuard Dashboard")
- [ ] All headings have semantic levels (h1, h2, h3)
- [ ] Form labels are associated (not just placeholder text)
- [ ] Button purposes are clear ("Pause Agent" not just "Pause")
- [ ] Live region updates are announced (agent log, balances)
- [ ] Data tables have proper headers (`<th scope="col">`)

### Colour Contrast

Verify WCAG AA compliance (4.5:1 for text, 3:1 for large text/components):

1. Use browser DevTools: `F12` > Inspect element
2. Check computed background and foreground colours
3. Use online contrast checker: https://webaim.org/resources/contrastchecker/
4. Focus on:
   - Text on background colours
   - Icon buttons (e.g., close, copy, pause)
   - Placeholder text in inputs
   - Disabled button states

### Focus Indicators

Every interactive element must have a visible focus ring:

1. Tap through page with `Tab`
2. All buttons, links, inputs should show a clear visual indicator (not just `:hover`)
3. Focus should never disappear (no `outline: none` without replacement)

## CI Integration

The accessibility test suite runs in CI:

```yaml
# .github/workflows/test.yml
- name: Run a11y tests
  run: cd dashboard && npm run test:a11y
```

Violations block merge. If a violation is acceptable (documented exemption), add to `dashboard/a11y.ignore.json`:

```json
{
  "violations": [
    {
      "impact": "minor",
      "rule": "color-contrast",
      "targets": ["button.disabled"],
      "justification": "Disabled state exempt per WCAG exception"
    }
  ]
}
```

## Common Issues & Fixes

| Issue | Fix |
|-------|-----|
| Icon button has no label | Add `aria-label="Button purpose"` |
| Link text is "click here" | Use semantic, descriptive text |
| Placeholder used as label | Add `<label>` with `for` attribute |
| Focus ring removed | Replace with `focus:outline focus:outline-2` |
| Colour is only indicator | Add icon, pattern, or text cue |
| Live updates not announced | Wrap in `<div role="status" aria-live="polite">` |
